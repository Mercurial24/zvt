# ZVT 交易体系架构梳理

本文档梳理项目中与**交易执行**相关的全部代码结构、数据流向和当前状态，作为后续实盘接入改造的参考基线。

---

## 1. 核心概念

| 概念 | 类 / 文件 | 职责 |
|------|-----------|------|
| **策略引擎** | `Trader` / `StockTrader` (`src/zvt/trader/trader.py`) | 时间轴驱动，运行 Factor 选股，产出买卖信号 |
| **交易信号** | `TradingSignal` (`src/zvt/trader/__init__.py`) | 策略引擎的输出，包含标的、方向、下单参数 |
| **信号监听器** | `TradingListener` (`src/zvt/trader/__init__.py`) | 接口，监听开盘/收盘/信号等生命周期事件 |
| **账户服务** | `AccountService` (`src/zvt/trader/__init__.py`) | `TradingListener` 子类，定义下单抽象接口 |
| **仿真账户** | `SimAccountService` (`src/zvt/trader/sim_account.py`) | `AccountService` 实现，K 线价格模拟撮合，结果写 SQLite |
| **QMT 账户** | `QmtStockAccount` (`src/zvt/broker/qmt/qmt_account.py`) | `AccountService` 实现，对接 QMT（xtquant）真实下单 |
| **远程代理** | `QMTForwardClient` (`src/zvt/broker/qmt/qmt_remote.py`) | HTTP RPC 客户端，在 Linux 上远程调用 Windows 的 xtquant |
| **RPC 服务器** | `server_windows.py` (`data_engine_qmt/server_windows.py`) | Windows 端 HTTP 服务器，接收 `/rpc` 请求执行 xtquant 操作 |

## 2. 交易信号（TradingSignal）

定义在 `src/zvt/trader/__init__.py`：

```python
class TradingSignal:
    entity_id: str               # 标的 ID，如 "stock_sz_000338"
    due_timestamp                 # 信号过期时间
    happen_timestamp              # 信号产生时间
    trading_level: IntervalLevel  # K 线级别
    trading_signal_type           # open_long / close_long / open_short / close_short
    position_pct: float           # 仓位比例（三选一）
    order_money: float            # 下单金额（三选一）
    order_amount: int             # 下单股数（三选一）
```

三个下单参数**必须且只能指定一个**，构造时会断言检查。

信号类型与委托方向的映射（`trading_signal_type_to_order_type`）：

| TradingSignalType | OrderType |
|---|---|
| `open_long` | `order_long`（买入） |
| `close_long` | `order_close_long`（卖出） |
| `open_short` | `order_short` |
| `close_short` | `order_close_short` |

## 3. 信号监听器生命周期（TradingListener）

```python
class TradingListener:
    def on_trading_open(self, timestamp):     # 每日开盘
    def on_trading_signals(self, signals):    # 收到交易信号
    def on_trading_close(self, timestamp):    # 每日收盘
    def on_trading_finish(self, timestamp):   # 全部运行结束
    def on_trading_error(self, timestamp, error):  # 异常
```

`AccountService` 在此基础上增加：

```python
class AccountService(TradingListener):
    def get_positions(self):
    def get_current_position(self, entity_id):
    def get_current_account(self):
    def order_by_position_pct(self, ...):   # 按仓位比例下单
    def order_by_money(self, ...):          # 按金额下单
    def order_by_amount(self, ...):         # 按股数下单
```

## 4. 策略主循环（Trader.run）

文件：`src/zvt/trader/trader.py`，`run()` 方法（约第 434 行）。

```
for timestamp in 交易时间序列:
    ┌─ 1. 跳过非交易日
    │
    ├─ 2. on_trading_open(timestamp)              ← 通知所有 listener
    │
    ├─ 3. 消费上一轮的 trading_signals             ← on_trading_signals()
    │     → 逐个 listener.on_trading_signals()
    │     → 清空 self.trading_signals
    │
    ├─ 4. Factor 更新，加入新标的
    │
    ├─ 5. 等待 K 线就绪（实盘模式下等到收盘）
    │
    ├─ 6. handle_factor_targets()                 ← 因子输出目标
    │
    ├─ 7. on_time(timestamp)                      ← 可重写的回调
    │
    ├─ 8. on_targets_selected_from_levels()       ← 多级别合成买卖名单
    │
    ├─ 9. on_profit_control()                     ← 止盈止损
    │
    ├─ 10. sell() / buy()                         ← 产生新的 TradingSignal
    │      → 追加到 self.trading_signals（下一轮消费）
    │
    └─ 11. on_trading_close(timestamp)            ← 通知所有 listener

on_finish(timestamp)                              ← 全部结束
```

**重要细节**：`buy()`/`sell()` 产生的信号在**下一个时间周期开始时**才被消费，这是为了避免用"未来数据"下单。

## 5. Trader.buy() / sell() 的信号生成逻辑

`buy()`（约第 266 行）：
- 过滤掉已持仓标的
- 计算仓位比例 = `long_position_control()` / 标的数量
- 生成 `TradingSignal(trading_signal_type=open_long, position_pct=...)`

`sell()`（约第 295 行）：
- 取当前持仓与待卖标的的交集
- 计算卖出比例 = `short_position_control()`
- 生成 `TradingSignal(trading_signal_type=close_long, position_pct=...)`

**注意**：默认路径下，信号始终使用 **`position_pct`** 参数。

## 6. 当前存在的三条交易路径

### 路径 A：策略回测（SimAccountService）

```
StockTrader.run()
  → Factor 选股 → buy()/sell()
  → TradingSignal(position_pct=...)
  → SimAccountService.handle_trading_signal()
    → get_kdata() 取 K 线收盘价
    → order_by_position_pct() 模拟撮合
    → 写 Order / AccountStats / Position 到 SQLite
```

- **可用**：回测和仿真模拟
- **构造绑定**：`Trader.__init__` 写死创建 `SimAccountService`（`trader.py` 第 95–106 行）

### 路径 B：QMT 账户直连（QmtStockAccount）

`QmtStockAccount` 有两套下单接口：

**接口 1 — 信号驱动（`handle_trading_signal`）**：

```
QmtStockAccount.on_trading_signals()
  → handle_trading_signal(signal)
    → xtdata.get_full_tick() 取实时行情
    → 买入用买一价，卖出用卖一价
    → order_by_amount(order_amount=signal.order_amount)
    → xt_trader.order_stock() 真实委托
```

**接口 2 — 参数驱动（`buy(BuyParameter)` / `sell(SellParameter)`）**：

```
QmtStockAccount.buy(BuyParameter)
  → 查账户资金 → 按权重分配金额
  → xtdata.get_full_tick() 取卖三价
  → 算手数（100 股整数倍）
  → xt_trader.order_stock() 市价委托

QmtStockAccount.sell(SellParameter)
  → 查持仓 → 按比例算可卖量
  → xt_trader.order_stock() 市价委托
```

### 路径 C：REST API 下单

```
POST /api/trading/buy  → qmt_context.qmt_account.buy(BuyParameter)
POST /api/trading/sell → qmt_context.qmt_account.sell(SellParameter)
```

- 在 `src/zvt/rest/trading.py` 中定义
- 调用路径 B 的接口 2（参数驱动）
- **限制**：代码中用 `platform.system() == "Windows"` 判断，Linux 上直接返回 500

## 8. 路径 A → 路径 B 的断裂点

要让策略引擎（`StockTrader`）的信号自动通过 QMT 下单，存在以下断裂：

### 断裂点 1：Trader 写死了 SimAccountService

```python
# src/zvt/trader/trader.py 第 95-106 行
self.account_service = SimAccountService(...)
self.register_trading_signal_listener(self.account_service)
```

构造函数中没有提供参数来替换 `account_service`，也没有"实盘模式"的开关。

### 断裂点 2：信号参数类型不匹配

| 产出方 | 参数 | 消费方 | 支持的参数 |
|--------|------|--------|------------|
| `Trader.buy()/sell()` | `position_pct` | `SimAccountService` | ✅ `position_pct` / `order_money` / `order_amount` |
| `Trader.buy()/sell()` | `position_pct` | `QmtStockAccount` | ❌ 只处理 `order_amount` |

`QmtStockAccount.handle_trading_signal()` 最终调用 `order_by_amount(order_amount=trading_signal.order_amount)`，但此时 `order_amount` 为 `None`。

### 断裂点 3：QmtStockAccount 缺少生命周期方法

`TradingListener` 要求实现的方法：

| 方法 | SimAccountService | QmtStockAccount |
|------|-------------------|-----------------|
| `on_trading_open` | ✅ 恢复账户状态 | ❌ 未实现（抛 NotImplementedError） |
| `on_trading_signals` | ✅ | ✅ |
| `on_trading_close` | ✅ 刷新持仓市值 | ❌ 未实现 |
| `on_trading_finish` | ✅ pass | ❌ 未实现 |
| `on_trading_error` | ✅ pass | ❌ 未实现 |

### 断裂点 4：REST API 平台限制

`rest/trading.py` 中 `buy`/`sell` 端点要求 `platform.system() == "Windows"`，Linux 环境下直接拒绝。但实际上 `QmtStockAccount` 在 Linux 下已能通过 HTTP RPC 远程下单。

## 10. 改造方案：三条路径统一

### 目标

- **回测模式**不受影响，默认行为与现在一致
- **实盘模式**下，策略信号自动流向 QMT 真实下单
- **REST API** 在 Linux 上也能用（通过 HTTP RPC 远程下单）

### 改动总览

| 步骤 | 文件 | 改动内容 |
|------|------|----------|
| 1 | `src/zvt/trader/trader.py` | `Trader.__init__` 增加 `account_service` 可选参数，支持外部注入 |
| 2 | `src/zvt/broker/qmt/qmt_account.py` | `handle_trading_signal` 支持 `position_pct`；补齐生命周期方法 |
| 3 | `src/zvt/rest/trading.py` | 去掉 `platform.system() == "Windows"` 限制 |
| 4 | `examples/trader/` | 新增实盘 Trader 示例 |

### 步骤 1：Trader 支持可插拔的 AccountService

当前 `Trader.__init__`（第 95–106 行）写死了 `SimAccountService`：

```python
self.account_service = SimAccountService(...)
self.register_trading_signal_listener(self.account_service)
```

改为：

```python
# 构造函数新增参数 account_service=None
if account_service is not None:
    self.account_service = account_service
else:
    self.account_service = SimAccountService(
        entity_schema=self.entity_schema,
        trader_name=self.trader_name,
        timestamp=self.start_timestamp,
        provider=self.provider,
        level=self.level,
        rich_mode=self.rich_mode,
        adjust_type=self.adjust_type,
        keep_history=self.keep_history,
    )
self.register_trading_signal_listener(self.account_service)
```

回测时不传参，行为不变；实盘时传入 `QmtStockAccount` 即可。

### 步骤 2：QmtStockAccount 补齐缺口

**缺口 1 — `handle_trading_signal` 支持 `position_pct`**

当前只读 `trading_signal.order_amount`，收到 `position_pct` 信号时 `order_amount` 为 `None` 导致下单失败。

改为：

```python
def handle_trading_signal(self, trading_signal):
    # ... 取实时行情、定价 ...
    if trading_signal.position_pct:
        acc = self.get_current_account()
        money = acc.cash * trading_signal.position_pct
        volume = int(money / price / 100) * 100
        if volume > 0:
            self.order_by_amount(..., order_amount=volume)
    elif trading_signal.order_money:
        volume = int(trading_signal.order_money / price / 100) * 100
        if volume > 0:
            self.order_by_amount(..., order_amount=volume)
    elif trading_signal.order_amount:
        self.order_by_amount(..., order_amount=trading_signal.order_amount)
```

**缺口 2 — 补齐生命周期方法**

`TradingListener` 要求的 `on_trading_open` / `on_trading_close` / `on_trading_finish` / `on_trading_error` 未实现，注册为 listener 后会抛 `NotImplementedError`。补齐为空实现：

```python
def on_trading_open(self, timestamp):
    pass

def on_trading_close(self, timestamp):
    pass

def on_trading_finish(self, timestamp):
    pass

def on_trading_error(self, timestamp, error):
    logger.error(f"trading error at {timestamp}: {error}")
```

## 11. QMT 回调系统的演变与替代方案

### 原始回调系统（已移除）

项目最早版本（commit `9358184`）中，`qmt_account.py` 包含一个完整的 `MyXtQuantTraderCallback` 类，继承自 xtquant 的 `XtQuantTraderCallback`，通过本地 socket 长连接接收 QMT 的交易事件推送：

| 回调方法 | 推送内容 |
|----------|----------|
| `on_connected` / `on_disconnected` | 连接状态变化 |
| `on_stock_order` | 委托回报（下单确认、状态变更） |
| `on_stock_trade` | 成交回报（实际成交通知） |
| `on_stock_asset` | 资金变动（余额变化） |
| `on_stock_position` | 持仓变动 |
| `on_order_error` | 委托失败 |
| `on_cancel_error` | 撤单失败 |
| `on_order_stock_async_response` | 异步下单回报 |
| `on_account_status` | 账号状态 |

注册方式：

```python
callback = MyXtQuantTraderCallback()
self.xt_trader.register_callback(callback)
self.xt_trader.subscribe(self.account)
```

在 commit `d90354e` 中被移除，原因：HTTP RPC 是请求-响应模式，无法承载 xtquant 的反向推送机制。

### 回调缺失的影响

没有回调系统意味着：

- **下单后不知道何时成交**：只能事后轮询 `query_stock_orders` 查委托状态
- **委托失败无感知**：价格不合适、资金不足等错误不会实时通知
- **无法做成交后的联动操作**：比如成交后更新本地持仓记录、触发下一步策略
- **日线级策略影响较小**：每天收盘后下单，次日检查即可；但分钟级/实时策略受影响大

### 替代方案对比

| 方案 | 实时性 | 复杂度 | 额外依赖 | 评估 |
|------|--------|--------|----------|------|
| **主动轮询** | ❌ 有延迟 | 低 | 无 | 当前现状，日线策略够用 |
| **Redis Pub/Sub** | ✅ 实时 | 中 | Redis（已有） | **推荐**，与行情推送架构统一 |
| **WebSocket** | ✅ 实时 | 高 | 需改 server_windows | 已有 Redis，没必要另建通道 |

### 推荐方案：Redis Pub/Sub 交易事件推送

与现有行情推送架构完全一致——`windows_market_pusher.py` 推行情到 Redis，这里推交易事件到 Redis，同一套模式。

#### Windows 端：回调 → Redis Publish

在 Windows 端注册 xtquant 原生回调，每个回调方法中将事件序列化后 publish 到 Redis 频道：

```python
import redis
import json
from xtquant.xttrader import XtQuantTraderCallback

class TradeCallbackPublisher(XtQuantTraderCallback):
    """将 QMT 交易回调事件发布到 Redis"""

    def __init__(self, redis_client, channel="zvt_trade_events"):
        self.r = redis_client
        self.channel = channel

    def _publish(self, event_type, data):
        msg = {"event": event_type, **data}
        self.r.publish(self.channel, json.dumps(msg))

    def on_stock_trade(self, trade):
        self._publish("trade", {
            "order_id": trade.order_id,
            "code": trade.stock_code,
            "volume": trade.traded_volume,
            "price": trade.traded_price,
            "time": trade.traded_time,
        })
```

```python
        elif event_type == "disconnected":
            logger.warning("QMT 连接断开")
```

## 12. 回测撮合机制分析与改进方案

### 当前撮合机制

#### 防止未来数据泄露

项目做了一个关键设计：`buy()`/`sell()` 在**当前周期末尾**产生 `TradingSignal`，但信号在**下一个周期开始时**才被消费执行（`trader.py` 第 456 行）。这保证了不会用"未来数据"做决策。

#### 当前取价逻辑

**回测（SimAccountService）**：

取 `happen_timestamp`（信号产生时间）对应 K 线的 **`close`（收盘价）** 作为成交价（`sim_account.py` 第 198–214 行），叠加固定滑点和手续费：

- 买入扣款：`order_amount × close × (1 + slippage + buy_cost)`
- 卖出到账：`order_amount × close × (1 - slippage - sell_cost)`
- 默认参数：`slippage=0.1%`、`buy_cost=0.1%`、`sell_cost=0.1%`

**实盘（QmtStockAccount）**：

| 方向 | 接口 | 取价 | 报价类型 |
|------|------|------|----------|
| 买入 | `handle_trading_signal` | 卖一价 `askPrice[0]` | 限价单（`price_type=11`） |
| 卖出 | `handle_trading_signal` | 买一价 `bidPrice[0]` | 限价单 |
| 买入 | `buy(BuyParameter)` | 卖三价 `askPrice[2]` | 市价单（`price_type=5`） |
| 卖出 | `sell(SellParameter)` | 不取价 | 市价单，`price=0` |

### 当前撮合的问题

| 问题 | 说明 | 影响程度 |
|------|------|----------|
| **日线用收盘价撮合** | 信号 T 日产生，用 T 日 `close` 在 T+1 日成交。A 股日线经常跳空高开/低开，用昨收撮合严重失真 | 🔴 高 |
| **不取整到手** | `order_amount` 是浮点数直接使用，不按 100 股（1 手）取整，可能出现买 0.3 股 | 🔴 高 |
| **手续费对称简化** | 买卖各 0.1%，但 A 股实际是：买入仅佣金（约万 2.5），卖出佣金 + 印花税（0.05%） | 🟡 中 |
| **100% 成交** | 下单必成，不考虑成交量限制，可能在日成交 100 万的股上买入 5000 万 | 🟡 中 |
| **无涨跌停判断** | 涨停也能买到、跌停也能卖出 | 🟡 中 |
| **固定滑点** | `slippage=0.1%` 不随股价、流动性或下单量变化 | 🟢 低 |

### 业界标准做法

成交价模型按复杂度递增：

| 级别 | 方法 | 原理 | 适用场景 |
|------|------|------|----------|
| L1 | 固定收盘价 + 固定滑点 | 当前项目的做法 | 极粗略估算 |
| **L2** | **下一根 K 线开盘价** | 信号 T 日产生，用 T+1 的 `open` 成交 | **日线策略标配** |
| L3 | 开盘价 + 动态滑点 | 市场冲击 = C × σ × √(V_order / V_daily) | 大资金量化团队 |
| L4 | VWAP / TWAP | 模拟日内分段执行，按成交量分布加权 | 机构级拆单执行 |
| L5 | L2 订单簿回放 | 用逐笔 Tick 数据模拟订单簿撮合 | 高频策略 |

Backtrader、Zipline、聚宽、米筐等主流框架默认使用 **L2（下一根开盘价）**。

### 改进方案

#### 第一档：必做（投入小，效果大）

##### 1.1 日线策略用下一根 K 线的开盘价

**问题**：当前 `handle_trading_signal` 用 `happen_timestamp`（信号产生日）的 `close` 撮合。对日线来说，这相当于用 T 日收盘价在 T+1 日成交，忽略了隔夜跳空。

**方案**：根据 K 线级别选择取价方式。

```python
def handle_trading_signal(self, trading_signal: TradingSignal):
    entity_id = trading_signal.entity_id
    happen_timestamp = trading_signal.happen_timestamp
    due_timestamp = trading_signal.due_timestamp
    order_type = trading_signal_type_to_order_type(trading_signal.trading_signal_type)
    trading_level = trading_signal.trading_level

    if trading_level >= IntervalLevel.LEVEL_1DAY:
        # 日线及以上：用 due_timestamp（下一根 K 线）的 open 价
        kdata = get_kdata(
            provider=self.provider, entity_id=entity_id,
            level=trading_level.value,
            start_timestamp=due_timestamp,
            end_timestamp=due_timestamp,
            limit=1, adjust_type=self.adjust_type,
        )
        the_price = kdata["open"].iloc[0] if pd_is_not_null(kdata) else None
    else:
        # 分钟线：跳空小，用 happen_timestamp 的 close 合理
        kdata = get_kdata(
            provider=self.provider, entity_id=entity_id,
            level=trading_level.value,
            start_timestamp=happen_timestamp,
            end_timestamp=happen_timestamp,
            limit=1, adjust_type=self.adjust_type,
        )
        the_price = kdata["close"].iloc[0] if pd_is_not_null(kdata) else None
```

**效果**：解决日线跳空高开/低开导致的撮合失真，这是回测真实性提升最大的一步。

##### 1.2 买入取整到 100 股（1 手）

**问题**：`cal_amount_by_position_pct` 和 `cal_amount_by_money` 算出的 `order_amount` 未取整。

**方案**：

```python

        want_pay = self.account.cash * order_position_pct
        order_amount = int(want_pay // cost // 100) * 100  # ← 取整到手
```

##### 1.3 手续费改为 A 股真实费率结构

**问题**：当前买卖各 0.1%，与 A 股实际费率差异较大。

**A 股实际费率**：

| 费用项 | 买入 | 卖出 | 说明 |
|--------|------|------|------|
| 佣金 | 万 2.5（0.025%） | 万 2.5（0.025%） | 可和券商商量，最低万 1 |
| 印花税 | 无 | 0.05% | 2023 年 8 月起降为 0.05% |
| 过户费 | 0.001% | 0.001% | 可忽略 |
| **合计** | **≈ 0.026%** | **≈ 0.076%** | — |

**方案**：调整默认参数：

当前 `buy_cost=sell_cost=0.001` 合计双边 0.2%，真实费率约 0.1%，虚高了一倍，会导致回测收益偏悲观。

##### 2.1 涨跌停检测

**问题**：涨停板封死时无法买入，跌停板封死时无法卖出，但回测中不做判断。

**方案**：取到成交价后，用 K 线数据近似判断涨跌停：

```python
def _is_limit_up(self, kdata_row):
    """近似判断：收盘 == 最高价 且 涨幅 >= 阈值"""
    if kdata_row["close"] == kdata_row["high"]:
        if kdata_row.get("pre_close") and kdata_row["pre_close"] > 0:
            change_pct = (kdata_row["close"] - kdata_row["pre_close"]) / kdata_row["pre_close"]
            return change_pct >= 0.095  # 主板10%，创业板/科创板需区分
    return False

def _is_limit_down(self, kdata_row):
    if kdata_row["close"] == kdata_row["low"]:
        if kdata_row.get("pre_close") and kdata_row["pre_close"] > 0:
            change_pct = (kdata_row["close"] - kdata_row["pre_close"]) / kdata_row["pre_close"]
            return change_pct <= -0.095
    return False
```

在 `handle_trading_signal` 中：

```python
if order_type == OrderType.order_long and self._is_limit_up(kdata_row):
    self.logger.warning(f"涨停无法买入: {entity_id}")
    return
if order_type == OrderType.order_close_long and self._is_limit_down(kdata_row):
    self.logger.warning(f"跌停无法卖出: {entity_id}")
    return
```

##### 2.2 成交量约束

**问题**：不限制成交量，可能在流动性极差的股票上下大单。

**方案**：单笔成交不超过该 K 线成交量的一定比例（默认 10%）：

```python
class SimAccountService(AccountService):
    def __init__(self, ..., volume_limit_pct=0.1):
        self.volume_limit_pct = volume_limit_pct

    def handle_trading_signal(self, trading_signal):
        # ... 取到 kdata 后 ...
        max_volume = kdata["volume"].iloc[0] * self.volume_limit_pct
        max_volume = int(max_volume // 100) * 100
        if order_amount > max_volume:
            self.logger.warning(
                f"成交量约束: {entity_id} 请求 {order_amount} 股, "
                f"限制为日成交量 {self.volume_limit_pct:.0%} = {max_volume} 股"
            )
            order_amount = max_volume
        if order_amount <= 0:
            return
```

##### 3.1 动态滑点模型

**问题**：固定 0.1% 滑点不反映真实的市场冲击。实际中，下单量越大、流动性越差，滑点越高。

**业界常用的平方根市场冲击模型**：

```
市场冲击 = C × σ × √(V_order / V_daily)
```

- `C`：冲击系数，通常经验值 0.1–0.5，需用历史数据校准
- `σ`：标的近期波动率（如 20 日收益率标准差）
- `V_order`：下单量
- `V_daily`：日均成交量

**方案**：

```python
def _dynamic_slippage(self, entity_id, order_amount, kdata):
    daily_volume = kdata["volume"].iloc[0]
    if daily_volume <= 0:
        return self.slippage

    closes = get_kdata(
        provider=self.provider, entity_id=entity_id,
        level="1d", end_timestamp=kdata["timestamp"].iloc[0],
        limit=20, adjust_type=self.adjust_type,
    )
    if len(closes) < 5:
        return self.slippage

    returns = closes["close"].pct_change().dropna()
    sigma = returns.std()
    participation = order_amount / daily_volume
    impact = 0.3 * sigma * (participation ** 0.5)
    return max(impact, self.slippage)
```

##### 3.2 分钟级模拟执行（VWAP/TWAP）

即使是日线策略，也可以用分钟线模拟日内执行过程。适用于大资金策略的精细化回测，需要有分钟级 K 线数据支持。

**VWAP 思路**：按日内分钟成交量分布，将大单拆分到各时段执行，最终成交价为各时段成交价的加权平均。

**TWAP 思路**：将大单均匀分到 N 个时间段执行，每段取该分钟的收盘价，最终成交价为简单平均。

此方案需要分钟级数据支持，实现复杂度高，建议在第一、二档完成后再考虑。
