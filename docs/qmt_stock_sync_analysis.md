# QMT 股票池同步问题排查与避坑指南

本文档总结了我们在将 QMT (券商通道) 与 ZVT 系统集成时，在**全量同步并构建本地基础股票代码库（MetaData）**过程中遇到的一系列“诡异”问题。通过对照实验和逐步排查，我们摸清了 QMT 服务端底层数据的边界用例，并最终实现了一套高度健壮的股票元数据入库方案。

---

## 阶段一：发现问题与原始架构痛点

**背景描述：**
原始的 ZVT 系统在 `qmt_quote.py` 中拼接股票列表时，采用了一种“混搭”模式：由于早期的 QMT API 在北交所（京市A股）板块的支持上有缺失，原代码在调用 QMT 接口获取 `沪深A股` 后，会**强制借助本地已经用东方财富（EM）通道落库的数据库**，查询北交所股票名单并拼接。

**痛点：**
这种设计使得底层交易逻辑模块强依赖上一级的另一个数据通道（EM 或 XYSZ），极大破坏了单一数据源的独立性与纯净度。不仅容易因为跨表导致找不到数据，还违背了量化框架解耦的设计原则。

**初步重构动作：**
切断对 EM 的第三方数据库依赖查询。全面采用 QMT 最新的原生板块查询：
```python
stock_list = xtdata.get_stock_list_in_sector("沪深A股")
stock_list += xtdata.get_stock_list_in_sector("京市A股")
```

---

## 阶段二：诡异的数量对不齐引出报错调查

全面切到原生 QMT API 后，通过交叉验证发现两个数据库中存在的严重偏差：XYSZ（互联网大厂接口集合）总共获得了 **5488** 只股票，而 QMT 的 `stock_meta.db` 库里最后却只存入了 **5190** 只股票。相差了整整三百多只！

### 调查实验：真的是数据丢失吗？
我们编写了一个验证对比脚本 `check_qmt_stocks.py`，直接调用 QMT API 获取原生代码列表，发现了惊人的事实：
* **原生 API 返回数**: `5504` 只
* **实际存进本地 DB 数**: `5190` 只

**为什么那三百多只（主要集中在新号段 920 开头的最新北交所股票）在代码遍历入库时神秘失踪了？**

### 报错溯源：
并非在持久存储时被静默丢弃，而是**整个录入程序崩溃了**。
当 QMT 返回这 5504 只名单时，它底层对于其中处于尚未上市或者极不活跃的新股（例如北交所的 314 只最新股票），其内部 `OpenDate`（上市时间）字段传出的是一个非法异常值：`0`。
我们在利用 `pandas.Timestamp` 进行解析赋值时 `to_pd_timestamp(stock_detail["OpenDate"])`：遇到了 `pandas._libs.tslibs.parsing.DateParseError: day is out of range for month`。
由于没有加上异常保护，当迭代到这种异常新股时，整个 `qmt_stock_meta_recorder.py` 直接抛错中断死亡！而我们能在 DB 中查到的 5190 条，其实是上一次没带这些毒新股在数据库中残留下来的历史快照。

### 解决方案：
增加并捕获 `OpenDate` 转换的异常，使用计算机开端纪元时间作为兜底方案。
```python
try:
    list_date = to_pd_timestamp(stock_detail["OpenDate"])
except:
    list_date = pd.Timestamp("1970-01-01")  # 若QMT乱发日期，就降级占位保命
```
加上安全垫后运行，最终 ZVT 将 5504 只代码完整落盘通过。

## 阶段三：为什么 QMT (5504) 反而比 XYSZ (5488) 多？

在我们把因为报错而没录入的股票重新塞回池子之后，QMT 的总数变成了 5504，相比东方财富数据源的 5488，反而溢出了！

### 实验与比对：
我们对两个 SQLite 数据库做了一个 `left out join`。找到了仅在 QMT 存在的 19 只孤本。
真相是：**QMT 的 `京市A股` 板块内有“内鬼”**。
QMT 并不严谨，它在返回的京市股票列表中夹带了大量的北交所**新发债券**（如：`821016` "25京投02"）以及**纯指数代码**（如：`899050` "北证50"）。这与我们“获取纯 A 股列表以便于做均等计算或下载统一K线结构”的底层目的背道而驰。

### 解决方案（防火墙拦截）：
由于如果不拦截这些“伪股票”，在后续批量向券商服务器请求行情数据或者下载财务表格时会继续报错，于是我们在最初的拉取层加设了严厉的正则表达式或数字前缀过滤，只允许主板、科创板、创业板及正常的北交所股票放行：
```python
# 沪指: 60, 68   深指: 00, 30.  北交所: 43, 83, 87, 92
valid_prefixes = ('60', '68', '00', '30', '43', '83', '87', '92')
stock_list = [s for s in stock_list if s.split('.')[0].startswith(valid_prefixes)]
```
加上防火墙再并用 SQL 去除了原数据库里的这 19 条残渣后，**得到最终 QMT 股票验证池数量：5485。**

## 阶段四：验证 XYSZ 多出的极少数股票（实操测试）

清除了杂质之后我们终于看清了真实账面：`(QMT) 5485 对比 (XYSZ) 5488`。
发现那唯一的 3 只“孤品”分别为：
`920168 通宝光电`, `920187 N通领` 和 `301680 N固德`。

发现它们清一色的特征：属于非常新的、本周刚挂牌申购甚至是在上市新股首日的标的。由于新浪财经和东方财富这种互联网大厂消息极其敏锐，立刻就划进去了；但是作为提供严苛量化通道后台的 QMT，为什么在板块函数中隐藏了它们？难道是我们的过滤或者请求有问题？

### Windows 直连排查实验：
编写了一个能直连 MiniQMT 底层的终端探测脚本并在 Windows 环境运行强下：
1. `get_stock_list_in_sector("沪深A股")` / `"京市A股"`：**查无此人。**
2. `get_instrument_detail()`：能拉到名字，因为其内部产品字典库（非股票行情库）已更新。
3. **强制索要 K 线（The Kill Switch）**：对其显式执行 `xtdata.download_history_data` 和 `xtdata.get_market_data(period='1d')`，服务器返回 **无情的是空字典 / None**。

### 最终定论与架构思考：
实验证明了：**QMT 不在代码列表中返回它们，并非漏报，而是尽职拦截。** 因为这类尚未正式有连续多日 K 线、数据并不牢靠的刚打新股，服务器后台还拿不出能跑量化回测的行情包，提前加进名单会导致全量下载脚本由于抓不到数据频繁报错而拥堵。

从自动化量化交易系统的角度来看，**绝对不应该越位使用第三方互联网名单代替券商实盘通道的名单做字典**。盲目增加池子却不能买卖只会引发更多雪崩。最好的架构决策依然是：
`100% 只信任 QMT 抛给我们的官方列表数据，配合我们自己布控的前缀垃圾分类防火墙`，才能营造出一个坚不可摧的最优 A 股标的池。
