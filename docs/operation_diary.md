# Quant系统维护与运维日记

此文档用于记录系统环境、数据模块的核心改动和运维操作，方便日后追溯和排查。如有新的重要操作，会作为新的日记条目追加于此。

---

## 2026-03-21

### 1. 消除 QMT 全量拉取时的 OOM 与失联死机
- **痛点**：由于配置了 8GB 的大空间 Swap，导致 Python 脚本在发生超额内存请求（超大体量 `pd.concat` 时）不会立刻被 OOM Killer 杀死，而是陷入了严重的 Swap 硬盘交换泥潭（Thrashing），使得整台服务器系统 I/O 堵死，SSH 终端响应断连，形同死机。
- **治理**：完全重构了 `scripts/daily_update_qmt.py` 里的归档函数 `finalize_yearly`。由内存极易溢出的一体化全表聚合，修改为了内存“恒定”的流式处理：逐个加载切片、匹配分发到年度文件写入后立刻执行 `gc.collect()` 回收。就算同步 30 多年几千万行数据，内存也死死钉在~200MB以下。

### 2. 统一 QMT K线数据为年度 Parquet 文件
- **逻辑改动**：将增量和全量的 QMT 原版日线/后复权日线改成了和 `xysz` 一样的**按年单文件存档**，极大地提升了以 `code` 维度拉长线检索的速度。
- **脚本发布**：编写且已执行了 `scripts/migrate_qmt_klines_to_yearly.py` 数据迁移脚本。为了绕开 PyArrow 解析含有 `date=XXXX` 分区目录时对内部强类型校验非常苛刻从而引发 `ArrowTypeError` 崩溃的问题，特别采用 `pd.read_parquet` 逐个文件重构读取引擎。

### 3. 系统底层 Swap 换页调度优化 (防卡顿)
- 鉴于 `apt install earlyoom` 发现你机器上自带的 `mirrors.ivolces.com` 镜像源暂时无法进行外网域名解析（DNS无响应）。
- 立即切换到了**更原生的解决方案（方案 2）**：执行了 `sysctl vm.swappiness=10`，并已把配置永固在系统的 `/etc/sysctl.conf` 文件里。这直接告诉 Linux 内核未来**极大降低依赖硬盘 Swap 的频繁率**，尽可能维持在内存中解决问题。这样以后哪怕再遇到小规模资源紧张，也绝不会让系统突然卡死。
