## 整体架构概览

交易模块可以看成一个四层建筑：

```
┌─────────────────────────────────────────────┐
│  第4层：策略主循环 (Trader.run)               │  ← 大脑，驱动一切
├─────────────────────────────────────────────┤
│  第3层：信号与监听器 (TradingSignal/Listener) │  ← 神经系统，传递指令
├─────────────────────────────────────────────┤
│  第2层：账户服务 (AccountService)             │  ← 双手，执行下单
│    ├─ SimAccountService (回测沙盘)           │
│    └─ QmtStockAccount  (实盘 QMT)           │
├─────────────────────────────────────────────┤
│  第1层：数据模型与持久化                      │  ← 骨骼，定义数据结构
│    ├─ ORM schemas (Order/Position/Account)  │
│    ├─ Pydantic models                       │
│    └─ REST API                              │
└─────────────────────────────────────────────┘
```

### Step 1: 数据模型层 — 先搞清楚"棋盘上有什么棋子"


| 文件                                 | 读什么                                                      |
| ---------------------------------- | -------------------------------------------------------- |
| `src/zvt/trader/trader_schemas.py` | 4 个 ORM 表：`TraderInfo`、`AccountStats`、`Position`、`Order` |
| `src/zvt/trader/trader_models.py`  | Pydantic 版的 `PositionModel`、`AccountStatsModel`          |
| `src/zvt/common/trading_models.py` | 外部接口参数：`BuyParameter`、`SellParameter`、`TradingResult`    |
| `src/zvt/trader/errors.py`         | 5 个异常类                                                   |


**要点**：

- `AccountStats` 是账户每日快照（现金 + 持仓市值 + 盈亏率），通过 `positions` relationship 关联到多个 `Position`
- `Position` 记录单只股票的持仓（做多数量 `long_amount`、均价 `average_long_price`、可用量 `available_long`）
- `Order` 是每笔委托记录
- `trading_t` 字段：0 = T+0（买入当天可卖），1 = T+1（A 股规则，买入次日才可卖）

### Step 2: 信号与枚举 — "大脑发出什么样的指令"

**文件**：`src/zvt/trader/__init__.py`

**要点**：

```
TradingSignalType（策略意图）          OrderType（柜台动作）
  open_long   ──映射──→  order_long          买入
  close_long  ──映射──→  order_close_long    卖出
  open_short  ──映射──→  order_short         做空
  close_short ──映射──→  order_close_short   平空
```

`TradingSignal` 是核心数据包，关键字段：

- `entity_id`: 标的（如 `stock_sz_000338`）
- `happen_timestamp`: 信号产生时间（取价用这个时间）
- `due_timestamp`: 信号过期时间（= happen + 一个周期）
- 三选一参数：`position_pct` / `order_money` / `order_amount`（构造时断言只能传一个）

**默认路径下，`Trader.buy()`/`sell()` 永远只用 `position_pct`**。

### Step 3: 监听器接口 — "谁在听指令"

**文件**：`src/zvt/trader/__init__.py`（同 Step 2）

```
TradingListener（接口）
  ├─ on_trading_open()     # 每日开盘
  ├─ on_trading_signals()  # 收到信号 ← 最核心
  ├─ on_trading_close()    # 每日收盘
  ├─ on_trading_finish()   # 全部结束
  └─ on_trading_error()    # 异常

AccountService（继承 TradingListener，增加下单能力）
  ├─ get_positions() / get_current_position() / get_current_account()
  ├─ order_by_position_pct()  # 按仓位比例
  ├─ order_by_money()         # 按金额
  └─ order_by_amount()        # 按股数（最终都走这个）
```

这是**观察者模式**：Trader 是发布者，AccountService 是订阅者。

#### 观察者模式详解

`Trader` 内部维护一个监听器列表，初始化时将 `SimAccountService` 注册进去：

```python
self.trading_listeners.append(self.account_service)
```

`run()` 主循环每个周期依次向所有监听器"广播"事件：

```python
for listener in self.trading_listeners:
    listener.on_trading_open(timestamp)    # 开盘广播
    listener.on_trading_signals(signals)   # 信号广播 ← 最重要
    listener.on_trading_close(timestamp)   # 收盘广播
```

**为什么不让 Trader 直接调用 SimAccountService 的方法？**

因为监听器列表可以同时挂多个：

```python
trading_listeners = [
    SimAccountService(...),    # 回测账户
    QmtStockAccount(...),      # 实盘账户
    MyLoggerListener(...),     # 自定义日志监听器
]
```

Trader 完全不需要知道挂的是哪种账户，只管广播。这就是解耦——`Trader` 和 `AccountService` 之间没有硬依赖。

#### AccountService 里的 `pass` 方法是抽象接口

`AccountService` 是一个**抽象基类**，只定义"账户服务应该能干什么"的规范，不提供实现。子类必须覆写（override）这些方法：

```
AccountService（规范层，pass 占位）
    ├── SimAccountService   ← 回测实现：内存撮合 + 写 SQLite
    └── QmtStockAccount     ← 实盘实现：调用 QMT API 真实下单
```

两套实现对同一接口的填充方式完全不同：


| 方法                      | SimAccountService | QmtStockAccount                   |
| ----------------------- | ----------------- | --------------------------------- |
| `get_positions()`       | 返回内存中的持仓字典        | 调用 QMT API 查询真实持仓                 |
| `get_current_account()` | 返回内存中的 cash/资产    | 调用 QMT API 查询资金账户                 |
| `order_by_amount()`     | 内存撮合，写 SQLite     | 调用 `xt_trader.order_stock()` 真实下单 |


> **注意**：`TradingListener` 的方法用 `raise NotImplementedError`（子类不覆写会报错），而 `AccountService` 的方法用 `pass`（更宽松）。原因是 `order_by_*` 三个发单方法不是所有监听器都需要实现的——轻量级监听器（如日志器）可以只关心事件回调，无需具备下单能力。

### Step 4: 仿真账户 — "回测沙盘怎么撮合"

**文件**：`src/zvt/trader/sim_account.py`

这是最长也最重要的文件，建议分块读：

#### 4a. 初始化 (L46-88)

- 默认 100 万本金，手续费买卖各 0.1%，滑点 0.1%
- `rich_mode=True` 表示钱不够时自动追加 100 万（方便回测不被中断）；**正式验证策略时应设 `rich_mode=False`**，否则回测结果会比实际资金规模更乐观

#### 4b. on_trading_signals → handle_trading_signal (L183-255)

完整撮合流程：

```
Trader.run() 把信号压入队列
       │ （信号在下一周期开始时才消费，防止用未来数据作弊）
       ▼
on_trading_signals(signals)          遍历信号，逐个调用 handle_trading_signal
       │
       ▼
handle_trading_signal(signal)
  ① TradingSignalType → OrderType（策略意图 → 柜台动作）
  ② 查 happen_timestamp 对应的 K 线，取 close 价作为成交价
     （如果取不到 K 线或价格异常，打 warning 跳过，不报错）
  ③ 根据信号的下单规模参数类型分发：
       position_pct → order_by_position_pct()
                          └→ cal_amount_by_position_pct() 换算股数
                                └→ order_by_amount()
       order_money  → order_by_money()
                          └→ cal_amount_by_money() 换算股数
                                └→ order_by_amount()
       order_amount → order_by_amount()  直接用
       │
       ▼
order_by_amount()
  ① get_current_position()  取（或新建）该标的的持仓对象
  ② 合法性检查：多空互斥（有多仓不能开空，反之亦然）
  ③ 平仓时检查 available_long/short 是否足够
  ④ 调用 update_position()
       │
       ▼
update_position()                    真正修改账户状态
  【买入 order_long】
    need_money = 股数 × 价格 × (1 + slippage + buy_cost)
    cash -= need_money
    average_long_price 按加权均价公式更新（见 4c）
    long_amount += 股数
    trading_t==0 时：available_long += 股数（T+0 立即可卖）
    trading_t==1 时：available_long 不动（等收盘后解冻）

  【卖出 order_close_long】
    cash += 股数 × 价格 × (1 - slippage - sell_cost)
    available_long -= 股数
    long_amount    -= 股数
    average_long_price 不变（见 4c 均价说明）

  写一条 Order 记录到 SQLite
```

#### 4c. update_position (L368-457)

实际修改账户状态的核心：

**均价（average_long_price）的计算方式**

代码采用**加权均价法**（A 股券商标准），公式：

```
新均价 = (旧均价 × 旧股数 + 成交价 × 新买股数) / (旧股数 + 新买股数)
```

例：持有 100 股均价 10 元，再买 100 股 @ 12 元 → 新均价 = (100×10 + 100×12) / 200 = **11 元**。

**部分卖出后均价是否应该变化？——两种核算方法对比**

这里存在两套合法的核算逻辑，分别对应不同的场景需求：


| 方法              | 卖出后均价       | 谁在用        | 含义                   |
| --------------- | ----------- | ---------- | -------------------- |
| **加权均价法**（当前代码） | **不变**      | 大部分券商后台    | 当初买这批股花了多少钱，和之后的买卖无关 |
| **净投入法**        | **随盈亏动态调整** | 同花顺"持仓成本价" | 我在这只股票上还有多少净资金暴露     |


数值举例（忽略手续费）：

```
买入 100 股 @ 10元，总投入 = 1000元
─────────────────────────────────────────────────────
情景 A：盈利卖出 50 股 @ 12元（回收 600元）
  加权均价法：剩余 50 股均价 = 10元  （不变）
  净投入法：  剩余 50 股均价 = (1000 - 600) / 50 = 8元  ← 成本降低

情景 B：亏损卖出 50 股 @ 8元（回收 400元）
  加权均价法：剩余 50 股均价 = 10元  （不变）
  净投入法：  剩余 50 股均价 = (1000 - 400) / 50 = 12元 ← 成本升高
```

两种方法计算的**总盈亏（已实现 + 未实现）完全一样**，只是中间状态的呈现方式不同。

代码里的 `# FIXME:如果没卖完，重新计算均价` 描述的正是净投入法的逻辑。这并不是误判，只是当前代码选择了加权均价法，所以这行不需要实现。

**如果要换成净投入法（与同花顺对齐），改动如下：**

在 `sim_account.py` 的 `update_position` 方法中，找到 `order_close_long` 分支，在更新 `long_amount` 后加入均价重算：

```python
elif order_type == OrderType.order_close_long:
    proceeds = order_amount * current_price * (1 - self.slippage - self.sell_cost)
    self.account.cash += proceeds

    current_position.available_long -= order_amount
    current_position.long_amount -= order_amount

    # ↓ 净投入法：用卖出回收的资金"摊薄"剩余持仓成本
    if current_position.long_amount > 0:
        # 卖出前的总投入 = 旧均价 × 卖出前总股数
        original_total_cost = current_position.average_long_price * (
            current_position.long_amount + order_amount
        )
        # 剩余净投入 = 总投入 - 本次卖出回收
        net_remaining_cost = original_total_cost - proceeds
        # 新单股成本（极端盈利时可能为负，用 max 保护）
        current_position.average_long_price = max(0.0, net_remaining_cost / current_position.long_amount)
    else:
        current_position.average_long_price = 0
```

> 注意：改成净投入法后，`on_trading_close` 里的 `profit` 和 `profit_rate` 计算逻辑含义会随之变化（变成"剩余未实现盈亏"），但已实现盈亏已经体现在 `cash` 里了，总资产计算不受影响。

**T+1 限制实现方式**

买入时仅 `trading_t=0`（T+0）的标的立即增加 `available_long`；`trading_t=1`（A 股默认）须等到下一次 `on_trading_close` 才把 `available_long` 同步为 `long_amount`，变相实现买入次日才能卖的约束。

每笔操作都写一条 `Order` 记录到 SQLite。

#### 4d. on_trading_close (L257-332)

每日收盘结算：

1. 清除空仓（`long_amount == 0` 的持仓记录移除）
2. 遍历所有持仓，取**最新**收盘价（`end_timestamp=当日`，取最近一根日线）计算：
  - `position.value = long_amount × 收盘价`
  - `position.profit = (收盘价 - average_long_price) × long_amount`
  - `position.profit_rate = profit / (average_long_price × long_amount)`
3. 汇总账户：`all_value = 持仓市值 + cash`，计算总盈亏率
4. **解冻 T+1**：`available_long = long_amount`（今日买入的股票明日起可卖）
5. 写 `AccountStats` 快照到 SQLite

### Step 5: 策略主循环 — "大脑怎么运转"

**文件**：`src/zvt/trader/trader.py`

#### 5a. **init** (L29-135)

初始化做四件事：

1. **参数收集与时间轴准备**（L48-88）：校验必填参数，生成交易日历
2. **写死 SimAccountService**（L95-109）：这是架构文档提到的"断裂点 1"——构造函数没有提供参数替换 `account_service`，想接 QMT 实盘没有入口
3. `**init_factors()` 初始化因子**（L111-119）：基类返回空列表，留给子类覆写（模板方法模式）。注意 `start_timestamp` 往前推了 `pre_load_days`（默认 365 天），给均线等指标留出历史数据预热期
4. **多级别校验**（L127-134）：收集所有因子的时间级别，校验 Trader 的 `level` 必须等于最小级别

**为什么主循环必须用最小级别？**

因为时间精度取决于最小级别。假设你有 5 分钟因子和日线因子：

```
时间轴（5分钟步进）：9:30  9:35  9:40  ...  14:55  15:00
                      │                              │
                   5分钟因子每步都算              日线因子只在这里算
                                               （一根日K走完了）
```

主循环必须以 5 分钟为步进，否则 5 分钟因子的信号就丢失了。日线因子不需要每 5 分钟都算——代码里通过 `is_finished_kdata_timestamp` 判断"这根日 K 是否走完了"，只在完成时刻触发。

看 `handle_factor_targets`（L411-414）：

```python
for level in self.trading_level_asc:  # 从小到大遍历
    if self.entity_schema.is_finished_kdata_timestamp(timestamp=timestamp, level=level):
        # 只有当这个级别的K线"走完"时才计算
```

如果 Trader level 设成日线，但因子里有 5 分钟级别的，主循环每天只跑一步，5 分钟因子**一天只算一次**——所有日内信号全丢了。所以必须校验：Trader 的步进不能比最细的因子还粗。

> 这是工业界多时间框架分析（Multi-Timeframe Analysis）的标准做法：大级别定方向，小级别定时机。

#### 5b. run() 主循环 (L474-575)

`run()` 是整个交易系统的**心跳发生器**，按时间轴一步一步走，每一步做同样的六件事——开盘、执行遗留指令、因子选股、风控、签发新指令、收盘结算。

**完整流程图（一个周期内）**：

```
┌─ 入口：for timestamp in 时间序列 ──────────────────────────────┐
│                                                                │
│  非交易日？ ──Yes──→ continue                                   │
│      │No                                                       │
│      ▼                                                         │
│  步骤1: on_trading_open()        → 广播"开盘了"                 │
│      ▼                                                         │
│  步骤2: on_trading_signals()     → 消费【上一轮】的信号          │
│      ▼                            （SimAccount 真正撮合在这里）  │
│  [等待K线就绪]                    → 实盘才有用，回测直接跳过      │
│      ▼                                                         │
│  步骤3: handle_factor_targets()  → 因子算出看多/看空名单         │
│         on_time()                → 用户自定义钩子                │
│         on_targets_selected_from_levels() → 多级别合并           │
│      ▼                            （多取交集，空取并集）          │
│  步骤4: on_profit_control()      → 止盈止损，强制加入卖出名单    │
│      ▼                                                         │
│  步骤5: sell() → buy()           → 生成信号压入队列（下轮消费）   │
│      ▼                                                         │
│  步骤6: on_trading_close()       → 广播"收盘了"，结算写库        │
│                                                                │
└─ 下一个 timestamp ─────────────────────────────────────────────┘
      最后: on_finish() → 通知监听器结束 + 画资产曲线
```

**逐步骤说明**：

**步骤 1：开盘广播（L489-494）**

日线级别每个 timestamp 都算"开盘"（一天一步）；分钟级别只在 9:30 等开盘时刻才触发。`on_trading_open` 广播给所有监听器，`SimAccountService` 收到后会**从 SQLite 重新加载最新的 AccountStats 和所有 Position**（`load_account()`），确保内存状态和数据库一致。完整的日间节奏是：

```
收盘 on_trading_close → 写快照到 SQLite → 隔夜 → 开盘 on_trading_open → 从 SQLite 读回
```

> **注意**：回测第一天（`timestamp == start_timestamp`）直接 return，因为 `init_account()` 已经在构造时创建了初始账户，不需要 load。

**步骤 2：消费上一周期的信号（L496-503）**

`trading_signals` 里装的是**上一轮** `buy()`/`sell()` 压入的信号，不是本轮的。这是整个架构最精妙的设计：

```
Day T:  因子看完今日K线 → 发现金叉 → buy() 生成信号 → 压入队列
Day T+1: 步骤2 取出信号 → SimAccountService 用 T 日收盘价撮合
```

如果在 Day T 当天就执行，相当于"我看了今天的收盘价，然后回到今天开盘就买了"——这就是未来数据作弊。延迟一个周期消费，彻底杜绝。`on_trading_signals` 执行完后清空队列（L375: `self.trading_signals = []`）。

**等待 K 线就绪（L505-537）——仅实盘有效**

回测时所有 K 线都是历史数据，这段直接跳过。实盘时有两种等待方式：

- **日线实盘**：`time.sleep(600)` 循环等到晚 7 点。因为日 K 线要等收盘后数据供应商更新完才有
- **分钟线实盘**（`real_time=True`）：算出离下一根 K 线完成还差多少秒，调 `factor.move_on(timeout=...)` 让因子带超时地轮询数据源

所以不是 `run()` 自己去查 K 线，而是 `run()` 计算"要等多久"，把等待工作委托给 Factor。

**步骤 3：因子选股 + 多级别合并（L539-548）**

三步串联：

1. `handle_factor_targets()` — 各因子在自己级别完成时计算，输出缓存到 `level_map_long/short_targets`
2. `on_time()` — 用户钩子，子类可覆写做任何自定义逻辑
3. `on_targets_selected_from_levels()` — 将各级别的缓存合并成最终名单

默认合并逻辑：**做多取交集**（所有级别都看多才买），**做空取并集**（任一级别看空就卖）。这体现了"保守买入、积极止损"的理念。

> **设计评价**：默认的交集/并集合并比较粗糙，但 `on_targets_selected_from_levels` 是一个**可覆写钩子**（方法注释明确写了 `overwrite it for your custom logic`），用户可以在子类中替换为更精细的逻辑。工业界常见的替代方案：
>
>
> | 方法          | 思路                      |
> | ----------- | ----------------------- |
> | 交集/并集（当前默认） | 大级别和小级别都看多才买            |
> | 加权投票        | 各级别因子按权重打分，综合评分超阈值才入选   |
> | 优先级层叠       | 大级别有一票否决权，小级别只在大级别允许时生效 |
> | ML 融合       | 各因子输出作为特征，模型决定最终仓位      |
>

**步骤 4：止盈止损（L550-557）**

风控遍历持仓，利润率超过 `profit_threshold[0]`（默认 3 = 300%）或低于 `profit_threshold[1]`（默认 -0.3 = -30%）的标的，加入卖出名单。注意合并方式是**并集**：

```python
short_selected = list(set(short_selected) | set(passive_short))
```

因子选出的卖出名单和风控触发的卖出名单做并集——两个来源的卖出理由都被尊重，不会互相覆盖。

> **设计评价**：默认阈值比较极端（300% 止盈、30% 止损），实际策略中很少遇到 300% 盈利，而 30% 亏损又太宽松。缺少移动止损（Trailing Stop，从高点回撤 X% 就走）和时间止损（持仓超 N 天不涨就走）。同样是可覆写钩子，用户可在子类中实现更精细的风控。

**步骤 5：签发买卖令（L559-564）**

**先卖后买**——卖出后 `cash` 增加，后续买入有更多资金可用。`sell()` / `buy()` 不是真的下单，而是生成 `TradingSignal` 压入队列，等**下一轮**步骤 2 消费。

**步骤 6：收盘广播（L566-571）**

和步骤 1 对称。`SimAccountService.on_trading_close` 做 Step 4d 中讲过的全部工作：清空仓、刷持仓市值、计算盈亏、解冻 T+1、写 AccountStats 快照到 SQLite。

**循环结束：on_finish（L575）**

所有时间戳走完后调用 `on_trading_finish` 通知所有监听器"结束了"，给监听器一个收尾机会。虽然当前 `SimAccountService.on_trading_finish` 是 `pass`，但这个钩子为未来扩展预留了接口：


| 场景    | on_trading_finish 里可以做什么 |
| ----- | ------------------------ |
| 实盘账户  | 断开与券商 API 的连接，释放资源       |
| 日志监听器 | 输出最终统计报告，关闭日志文件          |
| 通知服务  | 发微信/邮件"回测跑完了，总收益 XX%"    |


有 `open` 就有 `close`，有 `start` 就有 `finish`——资源管理的标准模式。

#### 5c. buy() / sell() (L302-359) 及仓位控制

`buy()` 和 `sell()` 不是真正下单，而是"签发买卖令"——把决策翻译成 `TradingSignal` 信号包裹，压入队列等下一轮消费。

##### buy() 三步流程

**第一步：过滤已持仓（L303-313）**

```python
current_holdings = [
    position.entity_id
    for position in account.positions
    if position != None and position.available_long > 0
]
entity_ids = set(entity_ids) - set(current_holdings)  # 差集：只买没有的
```

因子选出 {A, B, C}，但 A 已持有 → 过滤后只买 {B, C}。

> **Day T 重复买入风险**：由于 T+1 冻结机制，Day T 当天新买的股票 `available_long=0`，在同一天的步骤 5 中 `buy()` 再次执行时不会被过滤，可能生成重复买入信号。不过该信号在 Day T+1 才消费，此时 `on_trading_open` 已从 SQLite 加载了正确持仓状态，`handle_trading_signal` 通过 `position_pct` 计算会发现目标仓位已到位，实际影响有限，但链路不干净。

**第二步：仓位控制（L316-317）**

```python
position_pct = self.long_position_control()           # 总仓位比例
position_pct = (1.0 / len(entity_ids)) * position_pct # 均分到每只股
```

`long_position_control()`（L265-277）实现了一个渐进加仓策略：


| 当前持仓数  | 总仓位比例 | 含义           |
| ------ | ----- | ------------ |
| 0（空仓）  | 20%   | 刚开始小心翼翼      |
| 1~10 只 | 50%   | 策略"跑顺了"，加大仓位 |
| >10 只  | 100%  | 全仓           |


数值举例——总资金 100 万，空仓，因子选出 5 只：

```
position_pct = 0.2（空仓，总仓位20%）
每只分到 = 0.2 / 5 = 0.04（4%）
每只用 100万 × 4% = 4万
```

**第三步：生成信号压入队列（L319-333）**

为每只股票生成 `TradingSignal(position_pct=...)` 压入 `self.trading_signals`，等下一轮步骤 2 消费。信号始终使用 `position_pct` 参数（这也是架构文档中断裂点 2 的根源——QMT 只读 `order_amount`）。

##### sell() 流程（L335-359）

结构和 `buy()` 对称，但有一个重要区别：

```python
shorted = set(current_holdings) & set(entity_ids)  # 交集，而非差集
```

`buy()` 用**差集**（要买的 - 已有的），`sell()` 用**交集**（要卖的 ∩ 实际持有的）——不能卖没有持仓的股票。

`short_position_control()` 直接返回 `1.0`（全卖），是可覆写钩子。

##### 整体数据流：从因子到信号

```
Factor.get_targets()            返回 ["stock_A", "stock_B", "stock_C"]
       ▼
on_factor_targets_filtered      过滤，最多保留10只
       ▼
update_targets_by_level         缓存到 level_map_long_targets
       ▼
on_targets_selected_from_levels 多级别交集 → long_selected = {"stock_B", "stock_C"}
       ▼
buy(entity_ids={"stock_B", "stock_C"})
  ├─ 过滤已持仓：stock_B 已有 → 剩 {"stock_C"}
  ├─ long_position_control() → 0.2（空仓阶段）
  ├─ position_pct = 0.2 / 1 = 0.2
  └─ TradingSignal(entity_id="stock_C", position_pct=0.2) → 压入队列
       ▼
  【下一周期步骤2】
       ▼
SimAccountService.handle_trading_signal()
  → 取K线收盘价 → order_by_position_pct(0.2) → 算股数 → update_position → 写SQLite
```

##### 设计评价：当前仓位管理的不足

当前的 `long_position_control` / `short_position_control` 只做了最原始的等权分配。量化投资不只是选股（Alpha），**仓位构建（Portfolio Construction）是独立且同等重要的一环**。业界通常把量化投资分三层：

```
┌─────────────────────────────────────┐
│  第一层：Alpha 信号（选什么）         │  ← 因子选股，当前项目做的
├─────────────────────────────────────┤
│  第二层：仓位构建（买多少）           │  ← 当前项目极其粗糙
├─────────────────────────────────────┤
│  第三层：执行优化（怎么买）           │  ← 滑点、市场冲击控制
└─────────────────────────────────────┘
```

**业界主流的仓位管理方法**（从简单到复杂）：


| 方法                 | 核心思想                               | 适用场景                |
| ------------------ | ---------------------------------- | ------------------- |
| 等权分配（当前）           | 每只股票分同样的钱                          | 最粗略                 |
| ATR 仓位法            | 用波动率反推仓位——波动大少买，波动小多买，让每笔交易的风险贡献相等 | 个人量化、CTA 策略最常用      |
| 风险平价（Risk Parity）  | 让每只股票对组合的风险贡献相等，权重反比于波动率           | 资产配置、桥水 All Weather |
| 均值-方差优化（Markowitz） | 在给定风险下最大化收益                        | 理论最优，但对参数极敏感        |
| Kelly 公式           | 从胜率和赔率推导最优下注比例                     | 胜率可估计的策略            |
| Black-Litterman    | Markowitz 改良版，以市场均衡权重为起点叠加主观观点     | 机构级组合               |


**ATR 仓位法详解**（最推荐入门的方法）：

"单笔风险"不是算出来的，而是自己预先设定的——"这笔交易如果亏了，最多亏总资金的百分之几"。完整计算链条：

```
第一步（你决定）：单笔风险比例 = 1%
  → 总资金 100万 × 1% = 1万（这笔交易最多亏1万）

第二步（市场告诉你）：stock_A 的 ATR(20) = 2元（平均每天波动2元）

第三步（设定止损距离）：止损距离 = 2 × ATR = 4元

第四步（反推仓位）：
  最大亏损 = 股数 × 止损距离 → 1万 = 股数 × 4元 → 股数 = 2500股

验证：买2500股，跌4元，亏 2500×4 = 1万 = 总资金的1%  ✓
```

核心思想是**从"我能亏多少"反推"我该买多少"**。高波动股止损宽→少买，低波动股止损窄→多买，每笔风险贡献被拉平。

**卖出/减仓策略**：

业界有句老话：**"会买的是徒弟，会卖的是师傅"**。卖出一点不比买入简单——卖早了踏空，卖晚了回吐利润。卖出策略分三大类：

**止损类（保护本金）**：


| 策略              | 做法                  |
| --------------- | ------------------- |
| 固定比例止损（当前 -30%） | 亏 X% 就走             |
| ATR 止损          | 买入价 - 2×ATR 设止损线    |
| 时间止损            | 持仓 N 天不涨就走（资金有机会成本） |


**止盈类（锁定利润）**：


| 策略                  | 做法                               |
| ------------------- | -------------------------------- |
| 固定目标止盈（当前 300%）     | 赚 X% 就走                          |
| 移动止盈（Trailing Stop） | 从最高点回撤 X% 就走——让利润奔跑              |
| 分批止盈                | 涨到目标 A 卖 1/3，目标 B 再卖 1/3，剩余用移动止损 |


移动止盈是趋势策略的核心，举例说明：

```
买入价: 10元
股价走势: 10 → 12 → 15 → 18 → 20 → 17 → 14
最高记录: 10   12   15   18   20   20   20
回撤止盈线(从高点-20%):
          8   9.6  12  14.4  16   16   16
                                        ↑ 14 < 16 → 触发止盈！
结果：买在10，最高到20，卖在16（锁住大部分利润，也容忍了合理回调）
```

**再平衡类（组合层面）**：


| 策略     | 做法               |
| ------ | ---------------- |
| 定期再平衡  | 每月/每季调回目标权重，超配的卖 |
| 信号反转卖出 | 买入因子变为负面信号时卖出    |

### Step 6: QMT 实盘账户 — "真金白银怎么下单"

**文件**：`src/zvt/broker/qmt/qmt_account.py`

`SimAccountService` 是在内存/SQLite 里模拟撮合，`QmtStockAccount` 则是真金白银——通过 HTTP RPC 远程调用 Windows 机器上的 QMT（迅投客户端）进行真实下单。两者都继承自 `AccountService`，是同一接口的两种实现：

```
AccountService（抽象接口）
  ├── SimAccountService   → 内存撮合，写 SQLite
  └── QmtStockAccount     → RPC → Windows QMT → 券商柜台
```

#### 6a. `__init__`（L26-47）—— 建立连接

| 参数 | 含义 |
|------|------|
| `path` | QMT 客户端的 userdata 路径，如 `D:\qmt\userdata_mini` |
| `account_id` | 券商资金账号，如 `"8885660116"` |
| `base_url` | Windows RPC 服务器地址（默认 `192.168.48.207:8000`） |

初始化做三件事：

1. 创建 RPC 客户端 `QMTForwardClient`
2. 远程实例化 QMT 的 `XtQuantTrader` 和 `StockAccount`（不是本地创建，而是发 HTTP POST 让 Windows 端创建，返回远程代理）
3. `xt_trader.start()` + `connect()`，后续所有方法调用透明转为 RPC 请求

对比 Sim：Sim 创建 SQLite session + 初始化账户；QMT 建立 RPC 连接 + 登录券商。

#### 6b. 查询方法（L49-57）

```python
get_positions()          → xt_trader.query_stock_positions()   # 查全部持仓
get_current_position()   → xt_trader.query_stock_position()    # 查单只持仓
get_current_account()    → xt_trader.query_stock_asset()       # 查资金（cash, total_asset）
```

全部是 RPC 调用，查的是**券商柜台的真实数据**。注意 `_to_qmt_code` 做了 ID 格式转换：ZVT 用 `stock_sz_000338`，QMT 用 `000338.SZ`。

对比 Sim：Sim 从内存 `self.account` 对象读；QMT 每次从券商实时查询。

#### 6c. `handle_trading_signal`（L84-112）—— 信号驱动路径 & 断裂点 2

这是 `Trader.run()` 主循环产生的 `TradingSignal` 的消费入口：

```python
def handle_trading_signal(self, trading_signal):
    # 1. 过期检查——信号超过 due_timestamp 就丢弃
    if now_pd_timestamp() > trading_signal.due_timestamp:
        return

    # 2. 取实时行情——RPC 调用 xtdata.get_full_tick()
    quote = xtdata.get_full_tick([qmt_code])

    # 3. 定价——买入用卖一价（对手价成交），卖出用买一价
    price = tick["askPrice"][0]  # 买入
    price = tick["bidPrice"][0]  # 卖出

    # 4.【断裂点2】直接读 order_amount，但 Trader 发的信号用的是 position_pct
    self.order_by_amount(order_amount=trading_signal.order_amount)  # ← None！
```

**断裂点 2 的本质**：`Trader.buy()/sell()` 生成的信号**永远**用 `position_pct`，但 QMT 只读 `order_amount`（此时为 `None`），下单必然失败。对比 `SimAccountService.handle_trading_signal`（L217-240），它三种参数都支持：

```python
# SimAccountService 的分支处理（QMT 缺少的）
if trading_signal.position_pct:
    self.order_by_position_pct(...)    # 查账户资金 → 算比例 → 算股数
elif trading_signal.order_money:
    self.order_by_money(...)           # 金额 → 算股数
elif trading_signal.order_amount:
    self.order_by_amount(...)          # 直接用股数
```

**过期检查的含义与 Bug**：

`due_timestamp = 信号产生时间 + 一个K线周期`。即日线信号有效期一天，5 分钟信号有效期 5 分钟。初衷是防止执行太旧的信号。但当前实现**在日线实盘下必定误杀正常信号**：

```
Day T（2024-01-15）:
  步骤5: buy() → due_timestamp = 2024-01-15 + 86400s = 2024-01-16 00:00:00（凌晨零点）

Day T+1（2024-01-16）:
  步骤2: 消费信号 → QmtStockAccount.handle_trading_signal()
         → now = 2024-01-16 10:00:00（哪怕是当天上午）
         → due = 2024-01-16 00:00:00（凌晨零点）
         → 10:00 > 00:00 → 判定过期，丢弃！ ← 正常信号被误杀
```

问题根源：`due_timestamp` 算的是"下一天零点"，但消费时哪怕是 Day T+1 的上午，`now` 也已超过零点。日线实盘中主循环还会等到晚 9 点才处理（`trader.py` L510-518 的 `current.hour >= 21` 等待逻辑），过期更严重。5 分钟线也容易卡在边界——信号 due = 9:35，执行时 now ≈ 9:35~9:36，网络稍有延迟就过期。

**回测无此问题**——回测走 `SimAccountService`，其 `handle_trading_signal` 没有过期检查，直接处理信号。过期检查是 QMT 自己加的。

**修复方案（推荐）：用循环时间替代墙上时钟**

过期检查的语义应该是"这个信号是不是跨了太多周期"，而不是"现在几点了"。把 `now_pd_timestamp()` 换成循环传入的 timestamp：

```python
class QmtStockAccount(AccountService):
    def on_trading_open(self, timestamp):
        """记录当前循环周期的时间（同时补齐了断裂点3的生命周期方法）"""
        self.current_timestamp = timestamp

    def handle_trading_signal(self, trading_signal):
        # 用循环时间判断，而非墙上时钟
        if self.current_timestamp > trading_signal.due_timestamp:
            logger.warning(
                f"signal expired: generated {trading_signal.happen_timestamp}, "
                f"due {trading_signal.due_timestamp}, current cycle {self.current_timestamp}"
            )
            return
        # ... 正常下单逻辑
```

这样无论实际在晚 9 点还是凌晨执行，只要循环还在 Day T+1 这一步，信号就不过期。只有循环走到 Day T+2 时，Day T 的信号才算过期。同时这也为 QMT 补齐了 `on_trading_open` 生命周期方法（断裂点 3 的一部分）。

#### 6d. `order_by_amount`（L59-75）—— 真实下单

```python
self.xt_trader.order_stock(
    stock_code=stock_code,
    order_type=23,      # 23=STOCK_BUY, 24=STOCK_SELL
    order_volume=order_amount,
    price_type=11,      # FIX_PRICE = 限价单
    price=order_price,
)
```

最终落地的真实委托。对比 Sim 的 `order_by_amount` 在内存里扣钱加持仓，这里是发 RPC 由 xtquant 向券商柜台发送真实委托。

#### 6e. `buy(BuyParameter)` / `sell(SellParameter)`（L114-167）—— 参数驱动路径

这是**独立于 TradingSignal 体系的另一套下单接口**，供 REST API 手动下单用：

```
信号驱动：Trader.run() → TradingSignal → handle_trading_signal() → QMT（策略自动触发）
参数驱动：用户/前端 → POST /api/trading/buy → buy(BuyParameter) → QMT（手动触发）
```

**`buy(BuyParameter)`**：查账户资金 → 按 `position_pct` 或 `money_to_use` 算可用金额 → 按 `weights` 分配到各股票（无权重则等分）→ 取卖三价 → 取整到 100 股 → 市价单下单

**`sell(SellParameter)`**：遍历各股票 → 查实际持仓 → 按 `sell_pcts` 比例算可卖量 → 市价单下单

两条路径对比：

| | 信号驱动（handle_trading_signal） | 参数驱动（buy/sell Parameter） |
|---|---|---|
| **触发方** | Trader.run() 策略引擎自动触发 | REST API / 人手动触发 |
| **输入** | TradingSignal（策略产物） | BuyParameter/SellParameter（请求体） |
| **定价** | 买一/卖一价，限价单 | 卖三价（买）/不指定（卖），市价单 |
| **取整** | 未取整 | `int(... / 100) * 100` ✓ |
| **多股票** | 每只单独一个信号 | 支持一次性多只，支持权重分配 |
| **状态** | 断裂（收不到 position_pct） | 可用 |

#### 设计讨论：信号参数类型的选择

`TradingSignal` 支持三种下单参数（`position_pct` / `order_money` / `order_amount`），策略层应该用哪个？

| 参数 | 含义 | 抽象程度 | 适用性 |
|------|------|---------|--------|
| `position_pct` | 用总资金的 X% 去买 | 高 | **最佳**——与资金规模无关，100 万和 1 亿的策略代码完全一样 |
| `order_money` | 用 X 万元去买 | 中 | 一般——写死金额，资金规模变了要改策略 |
| `order_amount` | 买 X 股 | 低 | 最差——和股价、资金都强绑定，几乎不可复用 |

**策略层描述的应该是"意图"（把 20% 资金投入），不是"操作"（买 2500 股）**。`position_pct → 股数` 的转换属于"仓位构建层"的职责，应放在 `handle_trading_signal`（AccountService 层），而非策略层。原因：

- AccountService 知道真实账户状态（资金、持仓）
- 信号在下一周期才消费，生成时取的价格已过期，执行时才应取最新价
- 策略代码不需要区分回测/实盘的价格源，换 AccountService 实现即可

#### qmt_remote.py 的精妙之处

- `QMTForwardClient.call()`: 发 HTTP POST 到 Windows 端 `/rpc`，支持 JSON 和 pickle 两种序列化
- `RemoteObject`: 代理模式，`__getattr_`_ 拦截所有方法调用，转为远程 RPC
- `xtdata = get_xtdata_proxy()`: 模块级实例，任何 `xtdata.xxx()` 调用都透明转发到 Windows

#### rest/trading.py 的平台限制

`buy`/`sell` 端点有平台判断（第 103/113 行），Linux 上直接返回 500。这是架构文档提到的"断裂点 4"，因为 `QmtStockAccount` 在远程模式下已经可以在 Linux 上工作。
